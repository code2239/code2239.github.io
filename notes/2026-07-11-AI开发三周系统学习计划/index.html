AI 开发三周系统学习计划

背景：材料科学 + 凝聚态物理本科/硕士，已能用 AI 做网站、搭工作流、调 MCP/hook/skill 卡点：非 AI 科班出身，感觉进入「深水区」——AI 写的代码出了 bug 读不懂、修不了 时间：每天 4 小时以上，共 21 天

🧠 核心认知：什么会变，什么不会

❌ 会变的东西（不需要死记）

  • 具体的模型名称（GPT-4 → Sonnet 4 → Gemini 3 → …）
  • 某个框架的 API（LangChain v1 → v2 → v3）
  • 某个向量数据库的用法（ChromaDB → pgvector → …）
  • 具体的 prompt 技巧

✅ 不变的东西（值得投入去学）

永恒模式本质为什么不变
LLM API 调用输入文本 → 输出文本任何模型都是这个接口
Agent 循环while True: 思考 → 行动 → 观察所有 Agent 框架都是这个循环
RAG 管道检索 → 注入上下文 → 生成知识增强的核心范式
结构化输出验证链LLM 输出 → Pydantic 校验 → 重试让 AI 可靠的必要机制
异步并发等 I/O 时先干别的网络请求的本质限制

你现在学的不是「怎么调某个 API」，而是 「怎么和 AI 一起构建系统」 的能力。只要人类还负责做架构决策，这个能力就永远有用。

🧱 四大能力块（Block）总览

三周 = 4 个能力块，按顺序推进：

  Block A ──→ Block B ──→ Block C ──→ Block D
  Python      LLM 系统     Agent 循环   系统架构
  工程能力     编程能力     构建能力     整合能力

每个块是独立的、完整的能力单元。块内按天推进。

🧱 Block A：Python 工程能力（6天）

为什么必须学

现状（C 状态）：
  AI 写的代码「好像能看懂，但让自己改就不确定」

出 bug 时：
  ❌ 以前：只能把整个代码扔回给 AI → AI 改完又有新 bug → 循环
  ✅ 学完后：能看类型错误、看日志、精准定位到具体行

A1 · 类型系统（2天）

只需要掌握这几个：

概念掌握标准
str int float bool None看函数签名立刻知道「输入什么、输出什么」
list[str] dict[str, int]从返回类型知道数据结构
str | None知道参数「可能有，也可能没有」
Pydantic BaseModel核心——把 LLM 的模糊输出变成精确结构

AI 使用方式升级：

❌ 以前：「帮我写个函数搜索笔记」
✅ 现在：「帮我写 search_notes(query: str, top_k: int=5) -> list[Note]，
          Note 用 Pydantic 定义」

验收：AI 给一段带类型注解的代码，你能逐行解释每一处在干什么。

A2 · 异步编程（2天）

只需要掌握 3 个核心模式：

场景必须会写的代码
并发调 N 个 APIresults = await asyncio.gather(*[fetch(url) for url in urls])
流式读取async for chunk in stream: — LLM streaming 的本质
超时保护async with asyncio.timeout(30): — 防止 Agent 卡死

异步不是「多线程」，是 「等 I/O 的时候先干别的」。LLM 调用主要就是等网络返回，所以异步是 AI 开发的基石。

验收：能手写一个异步并发调用多个 LLM 并处理结果的函数。

A3 · Pydantic + 验证链（2天）

核心作用：Pydantic 是你和 LLM 之间的「翻译官」。

LLM 返回文本：「用户张三，25岁」
         ↓
Pydantic 用模具去接：
  class User(BaseModel):
      name: str
      age: int
         ↓
  user = User.model_validate(data)
  如果 LLM 忘了写 age → 立刻报错 → 让 LLM 重试

Pydantic 是所有 AI 框架（LangChain、LlamaIndex、FastAPI）的底层基础。不是可选项，是必选项。

验收：能定义嵌套 Pydantic 模型，并用它来解析和验证 LLM 的结构化输出。

🧱 Block B：LLM 系统编程（5天）

为什么必须学

你不理解 LLM API 底层长什么样，所以遇到问题只能猜。
学完后，所有 SDK 在你眼里就是「帮你拼字符串的包装」。

B1 · API 协议（2天）

亲手做一件事：用 httpx 直接调一次 LLM API，不用任何 SDK。

你会亲眼看到：

  • 你发出去的是一个 messages 数组的 POST 请求
  • 你收到的是一串 data: 开头的 SSE 事件流

核心产出：手写一个 LLMClient 类：

class LLMClient:
    async def generate(self, messages) -> str
    async def generate_stream(self, messages) -> AsyncIterator[str]
    async def generate_with_tools(self, messages, tools) -> Response

验收：不用任何 SDK，能手写出一个能调 LLM API、支持 streaming 和 tool use 的客户端。

B2 · 结构化输出 + Tool Use（2天）

Tool Call 的本质（不是魔法）：
{
  "type": "tool_use",
  "name": "search_notes",
  "input": {"query": "Transformer", "top_k": 3}
}

你的代码要做的：if name == "search_notes": func(**input)

B3 · 上下文管理（1天）

核心问题：LLM 上下文窗口有限，对话太长会超限。

学会：

  • 计算 token（tiktoken）
  • 超出时丢弃最早消息
  • 什么时候触发摘要压缩

🧱 Block C：从零手写 Agent（4天）

为什么必须学

Agent = 让 LLM 自己决定「下一步干什么」，循环执行。
这是 AI 开发最重要的模式。

C1 · 最小 Agent（2天）

Agent 全部秘密就是这个循环：

async def run_agent(user_input: str, tools: list[Tool]) -> str:
    messages = [{"role": "user", "content": user_input}]

    for _ in range(MAX_ITERATIONS):
        response = await llm.generate(messages, tools)

        if response.has_tool_call:
            result = await execute_tool(response.tool_call)
            messages.append(response.message)
            messages.append({"role": "tool", "content": result})
        else:
            return response.content

    return "Agent 达到最大迭代次数"

所有复杂的 Agent 框架（LangGraph、AutoGen、CrewAI）都是在给这个循环加「管理功能」。

C2 · 能力扩展（2天）

能力本质
多轮记忆在 messages 数组里保留历史
长时记忆关键信息存向量数据库
错误重试try/except + 重新请求 LLM
并行 Agentasyncio.gather(agent_1(), agent_2())

验收：能写一个 Agent——用户提问 → Agent 自己决定调什么工具 → 返回最终回答。

🧱 Block D：RAG + MCP 系统架构（6天）

为什么必须学

C 的输出是「一个有推理能力的 Agent」
但它缺少「知识」——不知道你的笔记里写了什么

RAG = 给你的 Agent 装一个「你自己的知识库」
MCP = 让任何外部程序都能用你写的工具

D1 · Embedding + 向量搜索（2天）

一句话说清：
  "苹果" → [0.12, -0.34, 0.56, ...] (1536 维向量)
  "香蕉" → [0.14, -0.31, 0.52, ...] (相似的向量)
  "汽车" → [-0.22, 0.78, 0.11, ...] (不同的向量)

搜索 = 把你问题转成向量，找最近的邻居。

实操：把你 Obsidian 里 20 篇笔记存进 ChromaDB，写出一个能搜索的 RAG 系统。

D2 · RAG 完整流程（2天）

用户问：「我去年写的关于注意力的笔记在哪？」
→ 问题转成向量
→ 向量数据库搜索最相关的 3 个片段
→ 片段拼进 system prompt
→ LLM 基于上下文回答 → 附上引用来源

D3 · MCP（2天）

MCP = 把你的 Agent 能力「插拔化」

之前：Agent 只能调代码里写死的工具
之后：Agent 可连接任何 MCP Server（文件/数据库/浏览器…）
     其他人也能连接你写的 MCP Server

Capstone 项目：做一个 MCP Server，让你的 Agent 能通过 MCP 搜索 Obsidian 笔记。

验收：Agent 能通过 MCP 搜索笔记并回答相关问题。

  ✅ ~5,000 字学习笔记

⚡ 高效学习建议

  1. 每天第一件事：花 5 分钟 review 昨天的笔记
  2. 卡住超过 30 分钟：直接问 AI——把你的代码 + 错误信息发过去，问「我的理解是……为什么不对？」
  3. 写笔记：每天 30 分钟写学习笔记，用自己的话解释，不是抄教程
  4. 不要死磕：你不是在考试，目的是「理解」不是「记住」
  5. 休息节奏：45min 学习 → 5min 休息，每天运动不要停

💡 写在最后

你从材料科学 → 凝聚态物理 → AI 开发，这条路本质上是用你的科学思维（建模、抽象、系统思考）来理解一个新的领域。

你现在感觉的「深水区」，恰恰是因为你站在了从「用户」变成「开发者」的临界点上。跨过去，AI 就不再是「帮你写东西的工具」，而是「你的执行者」——你做架构决策，AI 去实现。

而你现在要学的，就是做架构决策的能力。

这个能力不会贬值。AI 越发展，它越值钱。

「Vibe Coding 的上限，是你 debug 的能力。」 —— 把这个上限推高，就是你这三周要做的事。
