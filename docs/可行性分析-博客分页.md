# 可行性分析：博客列表分页（每页 10 篇）

> 日期：2026-07-10 | 涉及范围：3-5 文件（中等规模）

---

## 1. 现状分析

### 1.1 受影响页面（4 个列表页）

| 页面 | 路由 | 当前内容量 | 当前策略 |
|------|------|-----------|---------|
| 首页 | `/` | 最新 6 篇 | `.slice(0, 6)` 固定截断 |
| 博客列表 | `/blog` | 全部 33 篇（8 个子目录） | `sortedPosts.map()` 全部渲染 |
| 日记列表 | `/notes` | 全部 4 篇 | `sortedNotes.map()` 全部渲染 |
| 项目列表 | `/projects` | 3 个入口 | 全量渲染 |
| 资源列表 | `/resources` | 3 个入口 | 全量渲染 |

### 1.2 核心冲突：客户端筛选 vs 分页

**当前筛选机制**以博客列表为代表：

```
filter-controller.js 的工作方式：
  1. 所有 .post-item 都在 DOM 中
  2. 用户选择标签 → 遍历所有 .post-item → display:none/"" 控制显隐
  3. URL searchParams 同步，支持分享筛选链接（如 /blog?tag=Claude）
```

分页一旦实施，DOM 中只有当前页的 10 篇文章。如果用户在第 1 页选择了标签 "Claude"，第 2 页不会有匹配文章的原因有两个：
- 第 2 页的文章没有载入 DOM，筛选逻辑看不到它们
- 即使用 URL 参数跨页传递，每一页都是独立的 Astro SSG 页面，无法做到"跨页收集匹配项再分页"

这就是 **SSG 分页与客户端筛选的根本矛盾**。

---

## 2. 候选方案

### 方案 A：Astro 原生分页（`paginate()`）— SSG 多页

**改动**：`/blog/index.astro` 使用 Astro 的 `paginate()` 生成 `/blog/1`、`/blog/2`... 每页独立 HTML。

| 优点 | 缺点 |
|------|------|
| 零 JS 依赖，纯 SSG | **筛选只能在当前页内生效**，用户体验割裂 |
| SEO 友好（每页独立 URL） | 筛选状态（标签/搜索）需在每个页面间手动传递 URL 参数 |
| Astro 内置，代码量少 | `/blog` 原有 URL 需要改为 `/blog/1` 或重定向 |
| 首屏加载快 | "没有找到匹配文章" 的语义变模糊——可能是该标签的文章在别的页 |

**适合**：内容量大（50+ 篇）、不依赖客户端筛选的场景。

### 方案 B：客户端分页（"加载更多" / 无限滚动）

**改动**：保持全部文章在构建时以 JSON 形式内嵌，用 JS 控制可见数量。

| 优点 | 缺点 |
|------|------|
| **筛选逻辑完全不受影响**（全部文章仍在 DOM/数据中） | 首屏需要加载所有文章的数据（尽管渲染只 10 篇） |
| 用户体验流畅，无页面跳转 | 纯 JS 方案，SEO 只能看到前 10 篇 |
| "点击加载更多" 按钮自然 | 文章多了之后数据体积增大（每篇约 500-2000 字） |

**适合**：中等规模（20-50 篇）、重视筛选体验的场景。

### 方案 C：URL 参数分页（混合，推荐）

**改动**：在方案 B 的基础上，将页码同步到 URL 参数。

```
/blog?tag=Claude&page=2
```

| 优点 | 缺点 |
|------|------|
| 保留客户端筛选的全部能力 | 仍需要 JS |
| URL 可分享、可书签 | SEO 仍受限（爬虫不一定执行 JS） |
| 筛选 + 分页状态统一在 URL | 初始加载数据量随文章增长 |

**核心流程**：
1. 服务端：仍构建全部文章，在 `<script>` 中内嵌一个 JSON 数组（只含标题/日期/摘要/标签/slug，不含正文）
2. 客户端：读取 URL → 根据 tag/page 参数筛选+分页 → 渲染当前页
3. "上一页/下一页" 按钮更新 URL 参数并滚动到顶部

---

## 3. 推荐方案：方案 C（URL 参数分页）

### 3.1 理由

1. **不破坏现有筛选系统**——`filter-controller.js` 的标签筛选 + 搜索逻辑可以完整保留，只需在 `updateVisibility()` 之后加一步分页计算
2. **URL 可分享**——`/blog?tag=Claude&page=2` 是一个完整的状态快照
3. **改动集中**——主要改动在 `/blog/index.astro` 和 `filter-controller.js`，不涉及 `BlogLayout.astro` 结构
4. **符合 CLAUDE.md 宪法的"轻量化"原则**——在现有代码上叠加，不大规模重构

### 3.2 不改动的部分

- `BlogLayout.astro`（侧边栏 + 布局骨架）——不动
- `PostFilter.astro`（侧边栏组合）——不动
- `CategoryNav.astro`、`TagFilter.astro`、`BlogSearch.astro`——不动

### 3.3 预估改动量

| 文件 | 改动程度 | 说明 |
|------|---------|------|
| `pages/blog/index.astro` | 中 | 引入分页状态，内嵌文章 JSON，渲染逻辑从全量改为按页切片 |
| `features/blog/filter-controller.js` | 中 | 新增分页控制逻辑（当前页、总页数、翻页按钮渲染） |
| `pages/index.astro` | 小 | 首页已有 `.slice(0, 6)`，不动或改为可配置 |
| `pages/notes/index.astro` | 可选 | 如果日记也要分页，改动类似但需处理日历筛选的交互 |
| `styles/` | 小 | 分页按钮组样式（使用现有 CSS 变量） |

**总文件数：3-5，属于宪法分级中的"中等规模"。**

### 3.4 风险

| 风险 | 严重度 | 缓解 |
|------|--------|------|
| 文章增长到 100+ 篇时 JSON 数据过大 | 低（短期内不太可能） | 可在 50 篇阈值时升级到 SSG 分页+服务端筛选 API |
| 分页与日历筛选（日记页）的交互复杂 | 低 | 日记量小（4 篇），暂时不需要分页 |
| SEO 爬虫看不到分页后的内容 | 中 | sitemap 仍列出所有文章独立页面（`/blog/xxx`），列表页 SEO 影响有限 |

## 4. 实施步骤

```
Phase 1：博客列表分页（核心）
  1. 修改 pages/blog/index.astro — 内嵌文章数据 JSON + 引入分页组件
  2. 修改 filter-controller.js — 添加分页逻辑
  3. 添加分页按钮样式
  4. npm run build 验证

Phase 2（可选）：扩展到其他列表
  5. 首页 — 可改为"查看全部 → 分页列表"
  6. 日记 — 按需分页（当前仅 4 篇，暂不需要）
  7. 项目/资源 — 按需分页（当前各 3 个，暂不需要）
```

---

## 5. 结论

**可行，推荐方案 C（URL 参数分页）**。

当前内容量（博客 33 篇）已超过 10 篇/页的阈值，将产生 4 页，分页需求真实存在。

核心设计决策已被确定：**保留客户端筛选 + 叠加上分页层**，而非用 SSG 分页替代客户端筛选。这样既保持了现有的即时筛选体验，又为内容增长做好了准备。
