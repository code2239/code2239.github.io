# Agent 编码原则总结

> 基于星游寰宇 (xingyouhuanyu-site) 项目的实践经验提炼。
> 2026-07-08

---

## 一、设计原则（为什么这样设计）

### 1. 最小能力原则（Rule of Least Power）
> 选择能完成任务的最弱技术。

- 交互用 vanilla JS，不用 React/Vue/Svelte
- 样式用 11 个 CSS 变量 + Tailwind，不引入 CSS-in-JS 或 UI 库
- 数据用 Markdown + frontmatter，不用 CMS 或数据库
- 部署用 GitHub Pages 静态托管，不用服务器

**核心理念**：每增加一个技术依赖，就增加了它的全部复杂度。弱技术的优势是稳定、不 EOL、零维护成本。

### 2. 单一真相来源（Single Source of Truth）
> 每个决策只在一个地方定义。

- 颜色 → `tokens.css` 的 `:root`，45 个组件引用
- 分类 → `categories.ts`，构建脚本 + 侧边栏 + 分类页共用
- 站点信息 → `config.ts`，SEO + 首页 + about 共用
- 内容结构 → `content.config.ts` 的 Zod schema，所有文章 + 构建校验共用
- 组件规范 → `CLAUDE.md`，所有开发者遵循

**核心理念**：改一个地方 = 全站生效。消除信息分裂。

### 3. 关注点分离（Separation of Concerns）
> 每一层只做一件事，不跨界。

```
pages/     → 路由 + 数据获取（不写 UI）
layouts/   → 页面骨架（不写业务逻辑）
components/→ 纯渲染（不获取数据）
features/  → 功能模块（自包含，可插拔）
lib/       → 纯逻辑函数（不含框架代码）
content/   → Markdown 内容（不含代码）
```

**核心理念**：修改一个关注点时，不需要理解其他关注点。

### 4. 约定优于配置（Convention over Configuration）
> 用一致的约定替代显式声明。

- 6 段式组件模板：`imports → Props → data → markup → style → script`
- 文件命名即路由：`pages/about.astro` → `/about`
- `max-w-content` 一个类承载全站内容区布局规范
- 新增文章 = 新建 `.md` 文件，不改任何组件

**核心理念**：消除决策疲劳。约定让"正确的事"成为默认。

### 5. 尽早失败（Fail Fast）
> 问题发现得越早，修复成本越低。

```
写作 → 构建 → 部署 → 用户
  │      │      │      │
  │   Zod schema    │      │
  │   validate 脚本  │      │
  │   TypeScript    │      │
  │      │      │      │
  ▼      ▼      ▼      ▼
 报错   报错   不部署  用户看不到错误
```

**核心理念**：不在用户面前暴露错误。

## 二、工作原则（怎么做）

### 6. 图谱优先（Graph-First）
> 在理解大型代码结构时，先读 graphify 的知识图谱，再深入具体文件。

- **适用场景**：代码审查、架构重构、跨模块功能开发、跨文件 bug 排查
- **不适用场景**：单文件小改动、新增文章
- **操作**：读取 `graphify-out/GRAPH_REPORT.md`，获取 God Nodes、社区结构、循环依赖
- **效果**：节省 ~60% 探索 token，发现代码阅读无法发现的隐藏连接

### 7. 先分析，后编码
> 在实现功能之前，先做可行性分析。

- 明确改动范围（几个文件、哪些模块）
- 确定技术选型（自己写 vs 外部库）
- 识别风险点与缓解方案
- 大改动输出到 `docs/` 目录

### 8. 改动规模控制
> 改动越大，确认越充分。

| 规模 | 文件数 | 流程 |
|------|--------|------|
| 小 | 1-2 | 直接改 |
| 中 | 3-5 | 口头确认方案 |
| 大 | 5+ 或架构变更 | 写可行性分析报告到 `docs/` |

## 三、技术约束（什么能做，什么不能做）

### 9. 引入外部依赖的判断标准

优先级顺序：
1. 项目已有能力能实现 → 不改动依赖
2. 纯 vanilla JS 开源库（MIT/Apache 协议）→ 可引入
3. 需要新框架 → **不可引入**

判断 checklist：
- [ ] 零框架依赖？
- [ ] MIT 或 Apache 2.0 协议？
- [ ] 体积合理（gzip < 50KB）？
- [ ] 能通过 CSS 变量适配项目设计系统？

### 10. 设计系统零容忍

- 颜色：只用 `var(--color-xxx)`，禁止硬编码
- 框架：只用 Tailwind 4 + 项目自定义 CSS，禁止引入 shadcn/ui、Bootstrap 等
- 客户端 JS：只用 vanilla JS 内联 `<script>`，禁止引入 React/Vue/Svelte

### 11. 内容与代码分离

- 新增/修改文章 → 只改 `src/content/` 下的 `.md` 文件
- 新增/修改功能 → 只改 `src/features/`、`src/components/`、`src/lib/`
- 两类改动分开提交，不混在同一个 commit

## 四、质量门禁（什么时候算完成）

### 12. 构建必过
> `npm run build` 通过是提交和部署的前提条件。

- 任何改动后必须执行 `npm run build`
- 构建失败时优先排查 Astro SSR 问题（如 `window`/`document` 不可用）
- 构建 = 静态检查的防火墙：Zod schema + TypeScript + 模板语法 + 分类校验

### 13. 提交规范

```
<type>: <description>

type: feat / fix / docs / style / refactor
一次提交只做一个逻辑改动
commit message 用英文
```

## 一句话总结

> **用最弱的技术、在最少的文件中、做最少的改动、在最早的阶段发现错误。**
