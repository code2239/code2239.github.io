场景：个人 Web 开发项目，客户机 ↔ 虚拟机双机协作。

一、为什么用 Git 管理项目

1.1 核心能力对比

能力没有 Git有 Git
多机同步手动 scp，容易覆盖错git pull 一键同步
回退Ctrl+Z 仅最近几步git reset 回到任意历史版本
撤销改动祈祷有备份git checkout -- 文件名 秒还原
改动追溯靠记忆git diff 精确到每一行
提交历史靠记忆git log + 提交信息记录每次原因
分支实验复制文件夹 项目-实验版git branch 开分支，满意合并，不满意删除
换机器/重装拷贝整个文件夹git clone → npm install，一分钟恢复
多人协作传 zip 包同一套流程，无需改变工作方式

1.2 现实场景举例

场景一：两台机器改冲突了

在虚拟机上改了 3 个文件，客户机上改了 2 个文件（冲突了）：

没有 Git：手动打开 5 个文件逐行对比，大概率漏掉东西
有 Git：  git pull → 提示冲突 → VS Code 一键选择保留哪边的改动

场景二：改完发现是错的

改了一个功能，上线后发现有问题：

没有 Git：从备份文件夹里翻找旧版本，祈祷还在
有 Git：  git revert <那次提交的ID> → 精确撤销，其他改动不受影响

1.3 Git 的本质

存档 + 时光机 + 同步器

对于客户机 ↔ 虚拟机两台机器开发的场景，Git 几乎是必需的，否则同步问题会非常痛苦。

二、核心命令

# 日常使用
git add .
git commit -m "描述你做了什么"
git push

# 同步远程
git pull

# 查看历史
git log

# 查看具体改了什么
git diff

# 回退到某个版本
git reset --hard <提交ID>

# 撤销单个文件的改动
git checkout -- 文件名

三、版本管理策略

3.1 Git 不强制任何规范

你可以随便提交、随便回退，不制定任何策略也能用。但养成几个低成本习惯，收益巨大。

3.2 三个推荐习惯

① 提交信息写清楚做了什么

# ❌ 不好
git commit -m "update"
git commit -m "修复"
git commit -m "改"

# ✅ 好
git commit -m "修复首页导航栏在移动端错位的问题"
git commit -m "添加用户登录的表单验证"

回头看 git log 时一眼就知道每个提交干了什么，回退时不抓瞎。

② 提交粒度小一点

一次提交改 20 个文件涉及 5 个功能 → 回退时拖泥带水 一次提交只做一个改动 → 回退精准，不影响其他

③ 用标签标记里程碑

# 标记一个稳定可用的版本
git tag v1.0 -m "首页+导航完成，已验证可部署"

# 出问题时精准回退到这个版本
git checkout v1.0

你不需要”制定版本更新策略”，但 git tag 相当于给某个提交贴个标签（“这个版本是好的”），回退时不用翻日志找提交 ID。

3.3 什么时候需要正式版本策略

情况需要什么
项目上线，有人在用至少用 git tag 标记发布版本
多人协作约定分支规则（main 稳定，开发在别的分支）
对用户展示版本号语义化版本：v1.2.3（主版本.次版本.补丁）
需同时维护旧版本分支策略 + 标签

3.4 个人项目总结

必须做：  git commit 写清楚改了什么  ← 免费，收益巨大
建议做：  用 git tag 标记稳定节点   ← 5 秒操作，回退时省半小时
不需要：  制定正式版本管理流程      ← 等项目上线有人说了再说

四、典型工作流

# 在虚拟机上开始一天的工作
git pull                    # 拉取客户机上最新的代码

# 写代码...

git add .                   # 暂存改动
git commit -m "完成用户登录功能"  # 提交，写清楚做了什么
git push                    # 推送到远程

# 里程碑节点打标签
git tag v0.1 -m "登录+注册完成"
git push --tags

# 切回客户机继续
git pull                    # 代码已同步过来了
客户机                          GitHub                          虚拟机
  │                               │                               │
  │── git push ──────────────────>│                               │
  │                               │<── git pull ──────────────────│
  │                               │                               │
  │<── git pull ──────────────────│                               │
  │                               │── git push ──────────────────>│
  │                               │                               │

核心原则：代码不改在共享文件夹里，改在本地磁盘上，用 Git 同步。
