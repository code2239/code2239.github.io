## 2. 底层技术栈

MetaGPT **不自研大模型**，而是作为**多智能体编排框架**，兼容调用多种大模型：

| 支持模型 | 类型 |
|----------|------|
| **OpenAI GPT-4o / GPT-4 / GPT-3.5** | 闭源商业模型 |
| **Anthropic Claude 3.5 / Claude 4 系列** | 闭源商业模型 |
| **DeepSeek-V2/V3** | 开源/商业模型 |
| **Qwen（通义千问）系列** | 开源/商业模型 |
| **GLM（智谱）系列** | 开源/商业模型 |
| **LLaMA 系列** | 开源模型 |
| **HuggingFace 社区模型** | 多模型支持 |

### 是否支持用户自选 API
**是**。用户可通过环境变量或配置文件自由指定使用哪个大模型的 API。v0.7.0 后支持**为不同角色分配不同的 LLM**（如产品经理用 GPT-4，工程师用 Claude，QA 用 DeepSeek）。

### 关键技术：元编程（Meta Programming）
MetaGPT 的"Meta"来自**元编程（Meta Programming）**——将软件开发的标准作业流程（SOP）编码为提示词序列，使 LLM 智能体按角色分工协作。这是一种元层次的系统设计方法，而非底层模型研发。

> **信息来源**：[MetaGPT 官方文档](https://docs.deepwisdom.ai)、[arXiv 论文 2308.00352](https://arxiv.org/abs/2308.00352)

## 4. 功能全景解析

### 主要功能
> 一句话需求输入 → 多智能体协作 → 自动输出 PRD、架构设计、任务拆分、完整代码、测试用例的**端到端软件开发框架**。

### 核心功能与关键技术

#### 4.1 多智能体 SOP 流水线（核心差异化技术）

**技术**：标准作业程序（SOP）代码化 + 角色专业化 + Pub/Sub 消息池

MetaGPT 模拟一个"虚拟软件公司"，包含 5 个核心角色：

| 角色 | 职责 | 关键产出 |
|------|------|---------|
| **产品经理（Alice）** | 撰写 PRD、用户故事、竞品分析 | 需求文档、用户故事 |
| **架构师（Bob）** | 系统设计、数据结构、API 定义 | 架构图、数据模型、接口规范 |
| **项目经理（Eve）** | 任务拆分、依赖管理 | 子任务列表、依赖图 |
| **工程师（Alex）** | 编码实现（支持多轮代码审查） | 可执行源代码 |
| **QA 工程师（Edward）** | 编写和执行测试，迭代调试 | 测试报告、Bug 报告 |

**协作流程**：`用户需求 → PM(PRD) → 架构师(设计) → PM(任务) → 工程师(代码) → QA(测试)`

角色间通信采用**发布-订阅（Pub/Sub）消息池机制**——Agent 只订阅与自己相关的消息类型。每个角色的输出是**结构化文档**（JSON + Markdown），而非自由自然语言对话，有效降低"传话游戏"式的信息失真和幻觉累积。

**参考**：[arXiv 2308.00352](https://arxiv.org/abs/2308.00352)、[DeepWiki 架构解析](https://deepwiki.com/FoundationAgents/MetaGPT/2-core-multi-agent-framework)

#### 4.2 可执行反馈机制（Executable Feedback）

**技术**：运行时自验证 + 自动调试循环（最多 3 次重试）

代码生成后，系统**实际运行单元测试**，根据测试结果决定是否进入调试迭代：

```
编写代码 → 执行测试 → 检查结果
  ├─ 通过 → 继续下一任务
  └─ 失败 → 自动调试 → 重试（最多 3 次）
```

这一机制弥补了纯代码审查无法发现运行时错误的缺陷。论文报告在 **MBPP** 基准上取得 **5.4% 的绝对提升**。

**参考**：[MetaGPT 论文实验部分](https://arxiv.org/abs/2308.00352)

#### 4.3 Data Interpreter（数据解释器）

**技术**：分层图规划 + 工具集成与进化 + 置信度验证（ACV）

Data Interpreter 是 MetaGPT 团队于 2024 年 3 月发布的专门面向**数据科学场景**的 LLM Agent，被称为"开源版 Devin"。

三大核心技术：
1. **分层图规划**：将有向无环图（DAG）用于任务分解，支持动态计划重规划
2. **工具集成与进化**：自动从执行经验中抽象工具函数并复用
3. **置信度验证（ACV）**：类似白盒测试的自动验证循环

性能数据（ML-Benchmark）：

| 框架 | 综合得分 |
|------|---------|
| **Data Interpreter** | **0.95** |
| AutoGen | 0.86 |
| OpenInterpreter | 0.77 |
| TaskWeaver | 0.69 |

在 ICR 和 SVPC 数据集上分别比 AutoGen 提高 **24.7%** 和 **21.2%**。在 MATH Level-5 困难问题上比 AutoGen 提高 **26%**。

**参考**：[Data Interpreter 论文 arXiv 2402.18679](https://arxiv.org/abs/2402.18679)、[KDnuggets 介绍](https://www.kdnuggets.com/metagpt-data-interpreter-open-source-llm-based-data-solutions)

#### 4.4 增量开发与代码审查

支持 `config.inc = True` 增量模式，避免长上下文溢出。工程师可选配代码审查动作，支持多轮代码改写迭代。

#### 4.5 赛马模式（Race Mode）—— Atoms 平台

并行调用多个大模型（GPT、Claude、DeepSeek、Qwen 等）生成不同方案，自动评分选择最优。Benchmark 成功率从 0.30 提升至 0.85，成本降低 80%。

## 5. 功能实现案例

### 案例：使用 MetaGPT 生成一个 Flappy Bird 游戏（全自动）

**使用者操作**：

在终端输入一行命令：

```bash
metagpt "Create a Flappy Bird game in Python with Pygame"
```

**MetaGPT 自动完成的协作流程**：

| 步骤 | 角色 | 产出 |
|:----:|------|------|
| 1 | 产品经理 | 分析游戏需求 → 输出 PRD（游戏机制、用户交互、评分规则） |
| 2 | 架构师 | 系统设计 → 输出类图、数据结构、文件结构 |
| 3 | 项目经理 | 任务拆分 → 将开发分解为 15+ 个子任务 |
| 4 | 工程师 | 编写完整代码 → `main.py`、`bird.py`、`pipe.py`、`game_manager.py` |
| 5 | QA 工程师 | 编写并执行测试 → 运行代码检查错误 → 自动调试修复 |

**最终交付物**：

```
/workspace/FlappyBird/
├── src/           # 全部 Python 源码（可直接运行）
├── tests/         # 测试用例
├── docs/          # PRD 文档、架构设计文档
├── requirements.txt  # 依赖清单
└── README.md      # 项目说明
```

**结果**：生成的可运行游戏在 MetaGPT 论文的 SoftwareDev 基准评测中获得 **4/4 的最高可执行性评分**（远高于 ChatDev 的 2/4 和 AutoGPT 的 1/4）。

> **信息来源**：[MetaGPT 论文 Table 5](https://arxiv.org/abs/2308.00352)
