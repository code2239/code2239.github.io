## 3.1 系统集成架构概述

### 整体选择：单进程管道 + 微服务就绪

MVP 阶段采用 **单进程 Python 管道**（基于 LangChain/LangGraph 编排），但在架构上预留微服务拆分点。理由：
- 7 个模块中的数据流天然是 **线性管线**（模块1→2→3 和 4→5→6→7），无需微服务的复杂路由
- LangGraph 的 `StateGraph` 已提供模块间的状态传递和条件分支
- 单进程避免了网络序列化开销和运维复杂度
- 当某模块成为性能瓶颈时（如模块2的 LLM API 调用），可独立拆分为微服务

### 数据流架构

```
┌─────────────────────────────────────────────────────────────────────┐
│  输入层                                                              │
│  PDF/DOCX/扫描件 ──→ [模块1: Docling] ──→ DoclingDocument (JSON)     │
└────────────────────────────────────┬────────────────────────────────┘
                                     │
                                     ▼
┌─────────────────────────────────────────────────────────────────────┐
│  理解层                                                              │
│  DoclingDocument ──→ [模块2: SciBERT + LLM-IE] ──→ Entities + Triples│
│  Methods 章节 ──→ [模块3: LLM Structured Output] ──→ Steps DAG       │
└────────────────────────────────────┬────────────────────────────────┘
                                     │
                                     ▼
┌─────────────────────────────────────────────────────────────────────┐
│  知识层                                                              │
│  Entities + Triples ──→ [模块4: Neo4j + LLMGraphTransformer] ──→ KG │
│  全量文档嵌入 ──→ [模块5: BERTopic + LLM] ──→ 聚类 + 缺口报告        │
└────────────────────────────────────┬────────────────────────────────┘
                                     │
                                     ▼
┌─────────────────────────────────────────────────────────────────────┐
│  创新层                                                              │
│  缺口报告 + KG 上下文 ──→ [模块6: Co-Scientist] ──→ 排序假设列表      │
│  假设列表 ──→ [模块7: MARS + Elo] ──→ 评估报告 + 最终排序            │
└─────────────────────────────────────────────────────────────────────┘
```

### 各模块间的数据格式

| 接口 | 上游模块 | 下游模块 | 数据格式 | 传输方式 |
|:---|:---|:---|:---|:---|
| **I1** | 模块1 Docling | 模块2 | `DoclingDocument` JSON (含 Markdown、段落列表、表格、bbox) | Python 内存对象 |
| **I2** | 模块2 NER+RE | 模块4 Neo4j | `List[Entity]` + `List[Relation]` JSON | Python → Cypher 写入 |
| **I3** | 模块1 Methods | 模块3 | Methods 章节 Markdown 文本 | 字符串切片 |
| **I4** | 模块3 DAG | 模块4 Neo4j | `DAG {nodes: [{id, label, tool, params}], edges: [{from, to, type}]}` | Python dict → Cypher |
| **I5** | 模块1 全文 | 模块5 | SciBERT 嵌入向量 (768-d) | NumPy 数组 → BERTopic |
| **I6** | 模块5 缺口 | 模块6 | `GapReport {questions, contradictions, gaps, trends}` JSON | Python dict |
| **I7** | 模块4 KG | 模块6 | Neo4j 查询结果 (实体列表 + 关系路径) | Python dict |
| **I8** | 模块6 假设 | 模块7 | `Hypothesis {id, statement, novelty, feasibility, evidence}` JSON | Python dict |
| **I9** | 模块7 评审 | 输出 | `ReviewReport {scores, comments, rank}` JSON | 文件 / API |

## 3.2 图数据库与 LLM 的协同设计

### 协同模式：**先存图再查图（Write-then-Query）**

```
┌──────────────────┐     Cypher 查询结果      ┌──────────────────┐
│   Neo4j          │ ──────────────────────→ │   LLM 生成层      │
│   知识图谱        │                          │   (Co-Scientist,  │
│   32B节点/关系    │ ←────────────────────── │    MARS)          │
└──────────────────┘     抽取结果写回          └──────────────────┘
```

**工作流：**

1. **写入阶段（模块2→模块4）**：LLM-IE 抽取的实体和关系通过 `LLMGraphTransformer` 写入 Neo4j。先对实体做消歧（基于 SciBERT 嵌入的余弦相似度合并重复实体），再执行 Cypher `MERGE` 写入。

2. **查询阶段（模块4→模块6/7）**：
   - **直接图查询**：`GraphCypherQAChain` 将自然语言转为 Cypher（如"列出所有抑制 TGF-β 的化合物"）
   - **上下文增强**：将查询结果的实体关系网络作为 LLM 提示的一部分，让 Co-Scientist 生成"基于已知关系的合理延伸"假设
   - **事实校验**：模块7评审假设时，查询 KG 验证假设中引用的实体和关系是否真实存在于文献中

3. **反馈阶段（模块7→模块4）**：通过评审验证的假设关系可写回 KG 作为"待验证假设"节点（Hypothesis Node），与已确认的文献关系用不同颜色的边区分。

### GraphRAG 的渐进启用

MVP 阶段**不启用** Microsoft GraphRAG 的全局索引（成本太高）。Phase 3 启动后，在 Neo4j 上叠加两层：
- **社区摘要层**：定期（如每周）对增长的 KG 执行 Leiden 聚类 + LLM 生成社区摘要
- **声明协变量层**：追踪不同论文对同一主张的支持/反对，自动标记矛盾点

## 3.3 微服务拆分预备

虽然 MVP 是单进程，但以下边界已预留为未来的微服务：

| 服务名 | 模块 | 拆分信号 | 通信协议 |
|:---|:---|:---|:---|
| `doc-parser` | 1 | GPU 利用率 > 80% | REST (JSON) |
| `ie-service` | 2 | LLM API 调用成为瓶颈 | gRPC (更紧凑) |
| `kg-service` | 4 | 图数据库读写分离需求 | Neo4j Bolt 协议 |
| `inspiration-engine` | 6+7 | 计算密集 + 异步任务 | 消息队列 (RabbitMQ) |

# 第五章：分阶段实施路线图

## Phase 1 — 最小可行产品（MVP）| 0–3 个月

### 研发目标
**可运行的端到端管线：PDF 输入 → 知识图谱查询 → 基础假设生成**

### 包含模块
- ✅ 模块1（Docling 文档解析）
- ✅ 模块2（SciBERT NER + LLM-IE 关系抽取）
- ✅ 模块4（Neo4j 知识图谱存储和查询）
- 🟡 模块5（BERTopic 聚类 + 基础缺口检测）
- 🟡 模块6（Co-Scientist 的 Generation + Ranking Agent 简化版）

### 核心交付物
1. CLI 命令：`research-os parse paper.pdf` → 打印 Markdown + 实体/关系 JSON
2. Web UI（Streamlit 原型）：上传 PDF → 查看抽取结果 → 在知识图谱中浏览实体关系
3. 基础假设生成：输入一个科学问题 → 基于已入图谱的知识输出 3-5 个假设方向

### 里程碑验证指标
- PDF 解析准确率 ≥ 0.85（以 Docling 基准为参照）
- 实体识别 F1 ≥ 0.85（生物医学领域，以 BC5CDR 测试集验证）
- 知识图谱查询响应 < 500ms（1000 篇论文规模）
- 假设生成的相关性（人工评估 ≥ 3/5 分）

### 硬件/API 成本估算
| 资源 | 说明 | 月费用 |
|:---|:---|:---|
| 开发 GPU 服务器 | 1× RTX 4090 (24GB) 云实例 | ~$600 |
| Neo4j 服务器 | 自托管 Community Ed. (8 vCPU, 32GB RAM) | ~$200 |
| LLM API (GPT-4o-mini) | 信息抽取 + 假设生成（~5,000 requests/月） | ~$30 |
| 总计 | | **~$830/月** |

# 第六章：风险、成本与资源评估

## 6.1 风险矩阵

### 风险1：技术成熟度风险 🔴 高影响 × 中概率

**描述：** 模块3（实验流程 DAG 提取）和模块5（认知压缩）的候选技术多处于学术论文阶段，开源可用性低。BioWorkflow 代码未发布，MCRAH-GIE 代码需联系作者。

**缓解策略：**
- **MVP 阶段不依赖未开源工具**——模块3 用 LLM Structured Output 自建，模块5 用成熟的 BERTopic
- 建立 fallback 机制：如自建管线的步骤恢复率 < 50%，则降级为仅做段落分类（research_workflow 的 F1=0.98 方案）
- 每季度重新扫描一次领域（订阅 arXiv cs.CL / q-bio 分类），一旦 BioWorkflow 或 MCRAH-GIE 开源立即评估

### 风险2：LLM 幻觉与不确定性风险 🔴 高影响 × 高概率

**描述：** 模块2（关系抽取）、模块3（DAG 生成）、模块6（假设生成）均依赖 LLM。LLM 可能：凭空捏造实体（幻觉）、输出不存在的引用、生成逻辑上不合理但表面可信的假设。

**缓解策略：**
- **证据溯源机制**——每个抽取的实体/关系必须附带原文中的提及位置（模块1 保留的 bbox 坐标和段落编号）
- **KG 事实校验**——LLM 生成的假设中引用的实体必须能在 Neo4j 中找到（模块7 评审前先验校验）
- **人类-in-the-loop**——对 Top-K 假设和评分 < 阈值的抽取结果做人工抽检（MVP 阶段建议 10% 抽检率）
- **声明协变量**（Phase 3）——自动追踪不同来源对同一主张的支持度，标记"孤立主张"（仅一篇论文支持的关系）

### 风险3：数据依赖风险 🟡 中影响 × 中概率

**描述：** 不同出版商的 PDF 排版差异巨大（双栏/单栏、扫描件/原生 PDF、中/英文混排），模块1 的 Docling 可能对某些格式解析失败。特定领域术语（如中医方剂名称、化学 IUPAC 命名）的 NER 识别率可能骤降。

**缓解策略：**
- **模块1 双引擎 fallback**——Docling 失败时自动切换 OpenDataLoader PDF (Apache 2.0)
- **模块2 渐进标注**——对目标领域的代表性论文（100 篇）进行人工实体标注，用 LoRA 微调 SciBERT
- **格式白名单**——MVP 阶段明确支持范围（如"原生 PDF、英文、单栏或标准双栏"），逐步扩展

### 风险4：工程集成风险 🟡 中影响 × 中概率

**描述：** 7 个模块涉及多个异构组件（Python/Java/Neo4j/LangChain），版本兼容性冲突风险高。Neo4j GPLv3 许可对 SaaS 化部署的限制。LangChain 生态版本迭代快（API 变动频繁）。

**缓解策略：**
- **统一容器化**——所有组件 Docker 化，用 docker-compose 锁定版本
- **LangChain 版本冻结**——使用 `langchain==0.3.x` 稳定分支，避免追踪最新版
- **许可证合规审查**——Phase 3 前完成法律审查：Neo4j GPLv3 是否触发分发条款？MinerU AGPL 是否影响整体许可证？
- **抽象层**——在 LangChain/Neo4j 上封装一层自有的 Python API，降低底层库变动的影响面

### 风险5：评分校准风险 🟡 中影响 × 高概率

**描述：** 模块7 的 AI 审稿存在系统性分数膨胀（~7.8 vs 人类 ~5.6），可能导致所有假设都得高分、失去区分度。

**缓解策略：**
- **MVP 校准实验**——用 20-30 篇已有 ICLR/NeurIPS 审稿结果的论文做标定，计算本系统 AI 审稿的偏差系数
- **相对排序替代绝对评分**——Elo 评级系统（模块6 已有）天然适合相对排序，避免绝对评分问题
- **定期人工校准**——每月邀请 1-2 位领域专家对 AI 评分最高的 10 个假设做独立评审，对比偏差趋势
